## 项目背景

钢结构工程图纸（DWG/DXF 格式）中，焊缝信息分散在各构件图中，需要人工汇总统计。本工具通过解析 DXF 文件内的 `WeldMark` 和 `Part` 块，自动提取每道焊缝的：

- 所属构件（如 BE018、CO007）
- 焊缝位置（Above / Below 箭头侧）
- 焊脚尺寸 hf（mm），CJP 坡口焊为 None
- 焊缝长度（mm）
- 连接零件对（如 `BE020/p175`）

输出文件：`焊缝统计_auto.xlsx`

## 核心脚本详解

### `weld_extractor.py`

#### 顶部配置（修改这里来适配新图纸）

```python
FOLDER   = r"c:\...\hanf"       # DXF 文件所在目录
OUTPUT   = os.path.join(FOLDER, "焊缝统计_auto.xlsx")
SCALE    = 10.0                  # 1 CAD 单位 = 10 mm
SNAP_TOL = 1.5                   # 焊缝箭头与零件线的捕捉容差（CAD 单位）
MAX_HF   = 20                    # hf 上限；超过此值视为板厚标注
LABEL_TIP_TOL = 8.0              # 引线端点匹配零件的容差（CAD 单位）
```

#### 主要函数

| 函数 | 说明 |
|------|------|
| `parse_weldmark(blk)` | 解析 WeldMark 块：提取 hf、CJP 标志、圆圈标识（双面焊）、`3 SIDES`/`2 SIDES` 标注、`TYP` 标志、箭头位置 |
| `get_part_lines(blk)` | 获取 Part 块中所有 LINE 实体的几何信息 |
| `find_all_labels(doc)` | 从 Mark 块的引线端点提取零件编号（如 p122）|

| `parse_bom(doc, comp)` | 解析 Unknown 块中的材料表，得到零件厚度/宽度/长度/数量 |
| `hf_from_thickness(t)` | 按板厚查标准最小填角焊脚尺寸 |
| `extract_welds(dxf_path)` | 单个 DXF 文件的完整提取逻辑（主函数）|

## DXF 结构约定

脚本依赖以下 DXF 块命名规则（由 Tekla/AutoCAD 导出）：

```
WeldMark-<ID> - <视图ID>   → 焊缝标注块
Part-<ID> - <视图ID>       → 零件几何块
Mark-<ID> - <视图ID>       → 零件编号引线块
Unknown-<ID>               → 材料表（BOM）块（无视图ID后缀）
```

同一 `视图ID` 的 WeldMark 和 Part 块属于同一个视图，脚本按视图分组处理。

## 关键参数与常量

### 几何容差

| 常量 | 值 | 含义 |
|------|----|------|
| `SNAP_TOL` | 1.5 CAD | 箭头端点到零件线的最大距离 |
| `LABEL_TIP_TOL` | 8.0 CAD | 引线端点匹配零件线的最大距离 |
| `MIN_EDGE` | 1.5 CAD | 3-SIDES 中忽略的退化短边（< 15 mm）|
| `ADJ_TOL` | `SNAP_TOL+0.5` | 3-SIDES 中判断零件边邻接的容差 |

### hf 修正（Sub-rule 3）

当 WM 标注的尺寸恰好等于板厚或腹板厚时，按标准表替换为最小填角尺寸：

```python
_HF_FROM_T = {6:5, 7:5, 8:6, 9:6, 10:7, 11:8, 12:8, 14:10, 16:10, 18:12, 20:12}
```

### TYP（典型焊缝）处理

WM 文本包含 `TYP` 时，表示图中只标了一次，但实际有多个对称实例。脚本会：

1. 统计**主视图**（Part 块最多的视图）中该零件的实例数 → `typ_multiplier`
2. 将焊缝行复制 `typ_multiplier` 份输出
3. 对于 3-SIDES TYP：`typ_multiplier ÷ len(gusset_names)`，避免与多筋板逻辑重复计数

### BOM 回退（comp/comp 情形）

当 WM 箭头落在构件本体（comp）自身线上导致两端零件都是 comp 时，脚本扫描 BOM 中的零件宽度，找到与焊缝几何长度最接近的非 comp 零件（容差 15%），并以该零件在 `part_number_map` 中的实例数确定输出行数。

## 算法流程

```
DXF 文件
  │
  ├─ parse_bom()           读取材料表 → part_dims, comp_dims
  │
  ├─ 按视图 ID 分组
  │    WeldMark 块 → wm_by_view
  │    Part 块    → part_by_view
  │
  ├─ find_all_labels()     解析引线 → 零件编号
  ├─ assign_labels_by_leader_tip()  → part_number_map
  │
  └─ 对每个视图中的每个 WeldMark：
       │
       ├─ parse_weldmark()   提取 hf/CJP/annotation/is_typ
       │
       ├─ [3-SIDES 分支]
       │    找最小非 comp 零件作为筋板 (gusset)
       │    枚举筋板所有邻接边 → edge_rows
       │    TYP 倍数 × edge_rows → results
       │
       └─ [普通 WM 分支]
            choose_weld_line() 确定焊缝零件和长度
            hf 修正（Sub-rule 3）
            haunch 端面长度修正
            BOM 宽度修正
            TYP 倍数 / BOM 回退倍数
            → results

  结果写入 Excel（焊缝统计_auto.xlsx）
```

## 已修复问题（按类别汇总）

### 碎片级邻接检测（BE023 p42 标签修复）

- **碎片邻接元数据**（`nb_start`/`nb_end`）：边收集时将每条碎片的起终点真实邻接块记录进 `g_ln` 字典，在 unlabeled passthru 的 remap 之前保存原始值
- **构件体识别**（~L860-870, ~L890-900）：邻接 remap 前检查未标签块是否有 LINE 长度匹配 `comp_dims`（翼缘宽/截面深），匹配则跳过 remap 保留为构件体标签 —— 防止 CO 剖面视图中 comp 被错误映射到非 comp 零件
- **双端边过滤**（~L901）：起终点接触不同 neighbor 的碎片（如 p42 的 73mm 边，一端触 BE023 一端触 p43）直接丢弃，不参与边合并
- **碎片级标签覆写**（~L1184-1202）：输出时扫描合并边的来源碎片 `nb_start`/`nb_end` 数据，仅当全部碎片都无 comp 接触时才覆写到非 comp
- **普适性**：所有 3-SIDES gusset 的边收集均记录碎片邻接；BE 系列因 comp 已有标签，前三项改动为零操作

### 圆圈标识（围焊/全焊）检测与 BOM 合成边生成

- **圆圈符号检测**（`parse_weldmark` ~L148-197）：收集 WeldMark 块中的 ARC 实体（两个半圆弧组成完整圆圈），圆心 y≈ref_y ±1.0、半径 1.0~2.5、同圆心 ARC≥2 判定为圆圈符号
- **双面焊覆写**（~L240-249）：圆圈替代上侧三角形 -> `has_above=True`，单侧标注尺寸复制到另一侧
- **BOM 合成边生成**（~L791-809）：圆圈板不枚举 DXF 几何边，改用构件截面维度 `[flange_w, weld_line, flange_w]` 生成三边全焊：
  - `flange_w` = `comp_dims['flange_w']`（构件翼缘宽）
  - `weld_line` = WM 箭头处焊缝线的 DXF 绘制长度
- **覆盖构件**：CO006 p33/p83（HN300×150→[150,232,150]×2面=12行）、CO007 p47（HW250×250→[250,172,250]×2面=6行）、CO008 p92（同上，6行）
- **构件体圆圈防护**：CO009/CO010 的圆圈标注在构件自身，仅启用双面焊，不触发边枚举（避免自引用边爆炸）

### BOM 解析与长度映射
- **BOM qty 修正**（`:506-531`）：按 X 坐标排序取第二个 `\d{1,2}` 数字（数量列），修复 qty 取到序号的问题
- **CO BOM 长度回退**（`:1127-1144`）：柱型构件剖面投影失真，geo→bw/bl 容差 35%（原 25%/8%），p124/p125 `90.5→116, 170→220`
- **p200 Strategy B**（`:836-855`）：排序盲配→接近度匹配，BE023 `268→268` 不再错配 `140`
- **p126 Strategy A 歧义守卫**（`:852-854`）：两 BOM 维度距离差 <8% 时跳过，避免 `90.5→110` 误修正
- **BOM 板长推断**（`:970-981`）：当 BOM 缺板长时从筋板几何中位值推算（BE022 p200→268）
- **BOM 重量过滤**（`:547-549`）：`bom_len > bw*4` 时丢弃（p200 重量 950 不再误判为板长）
- **Strategy B 宽映射守卫**（`:1065-1067`）：边→bw 超过 80% 差异时保持 geo（p42 33→73 阻止，p200 438.9→140 允许）
- **p26/p48 case1-skip**（`:1030-1034`）：geo=bl 时不再错误替换为 bw

### CJP / hf 修正
- **CJP 填角移动**（`:217-222`）：CJP+填角同侧时填角移至对侧
- **CJP groove 守卫**（`:1110-1114, :758-762`）：groove 存在时跳过板厚→hf 修正
- **双面同值守卫**（`:1155, :760`）：上下侧 hf 相同且 ≤20 时跳过修正（p125 `9→6` 误修）
- **非 BOM 零件 hf 回退**（`:1065-1072, :1314-1322, :1335-1343`）：用 `comp_web_t` 推算默认填角

### Part 标签与自引用
- **comp-backoff**（`:965-973`）：视图缺 comp Part 时，p124/p47→CO007/p47
- **无标签透传**（`:703-730`）：无标签 Part→最近已标签邻居（CO009 Part-3124→p16）
- **边缘级标记扫描**（`:794-838`）：替代原有透传逻辑，对无标签邻接直接用筋板边扫描最近已标签非 comp 零件
- **CO 自引用豁免**（`:932, :1160`）：仅 CO 构件的 comp/comp 标签保留

### 边数与去重
- **跨视图去重**：同标签+同长度跨视图时去重，仅 BOM qty=1 启用
- **自由边剔除**：合并后 >期望边数时按连接数剔除，仅 conn=0 丢弃
- **单端点过滤**：无标签+无透传的单端点边排除
- **邻接容差修正**（~L798-799 等 6 处）：`<`→`<=`，修复 CAD 间隙恰好等于 ADJ_TOL 时的邻接检测失败
- **合成边去重豁免**：BOM 合成边（圆圈板的三边）允许相同长度重复

### BE021 连接件后处理（comp-gusset 枚举） ✅ 已落地

- **触发条件**：3-SIDES WM 视图内，gusset 为 comp 体且 connected plate 无 WM 覆盖

## 已尝试方案与失败原因

### 方案 1：邻接焊缝生成（非 comp 零件默认有焊缝）

**思路**：对所有非 comp 零件，枚举其邻接边作为焊缝产出，跨视图去重过滤。

**结果**：
- 无守卫 → 609 行（CO010 爆增 204 行）
- 仅未覆盖零件守卫 → CO007 的 p100 因已有 `p100/p126` 一行被跳过

**失败原因**：守卫阈值无法设定——"有一行但不全"时该补几行缺乏依据，形成循环依赖。

### 方案 2：跨视图同厚板筋板 TYP 搜索

**思路**：3-SIDES TYP WM 的筋板厚度匹配同类型板件，在其它视图自动生成 3-SIDES 边。

**结果**：
- CO 守卫版本 → 361 行（CO007=61, CO008=53 均超计数）

**失败原因**：同厚板件（p124/p125/p127）本身已有普通 WM 标注，新增 3-SIDES 边与已有行重叠。**时序问题**：兄弟边生成时普通 WM 尚未处理，无法增量去重。

### 方案 3：边缘级标记扫描（替代透传）

**思路**：将透传逻辑（无标签零件→最近已标签零件的距离）替换为直接扫描筋板边到已标签非 comp 零件的距离。p100 距筋板边 7.5 CAD，应优于 CO007（无 Part 块）。

**结果**：317 行（CO007=37 未变，BE 全系列稳定）。

**失败原因**：CO007 视图 2424 中 CO007 的 Part 块实际距筋板边仅 2.7 CAD（更近于 p100 的 7.5）。comp→non-comp 覆写的距离比较条件 `_nc_d <= _comp_d` 不成立，p100 无法胜出。**几何层面死胡同**——图纸上 CO007 本体就是比 p100 更接近筋板边。

### 方案 4：双端邻接标签覆写（BE021 p122/p123）

**思路**：筋板边两端分别接触 p122 和 p123 时，输出 `p122/p123` 而非 `BE021/p122`。

**结果**：多次迭代未落地（边收集→合并→去重流水线均假设单邻接格式，改动 10+ 处但仅影响 4 行）。

**失败原因**：BE021 视图存在多个同标签 p122 Part 块，碎片合并后丢失两端邻接信息。扫描第二近标签时总先找到另一个 p122 而非 p123。**架构受限**——流水线深度耦合单邻接设计。

### 方案 5：BOM 取整统一化（round→int）

**思路**：将 `round(bw)` 全局替换为 `int(bw)`（截断），使 p125 116→115 与手册对齐。

**结果**：精确匹配从 134→114（丢失 20 条），手册取整无统一规则——部分构件用 round、部分用 int。

**失败原因**：`int(312.7)=312` vs `round(312.7)=313`，工程惯例有分歧。手册的约定随构件而异，全局替换得不偿失。

### 方案 6：Per-edge comp 距离恢复

**思路**：压缩→non-comp 覆写后，对每条边独立判断 comp 是否更近，恢复 comp 标签（CO009 p144 的 164.5mm 边应保留 CO009/p144）。

**结果**：restore-comp 过激，所有边被恢复为 comp 标签。CO009 p144 的 p143/p144 也丢失。

**失败原因**：碎片合并后丢失碎片→输出边的映射，per-edge 检查只能基于全局最小距离做一刀切判断。

### 方案 7：BE021 self-ref 抢救 + 非 comp 对标签覆写

**思路**：对 BE021/BE021（self-ref）边，扫描 gusset 碎片找最近非 comp 替换标签；对 comp/非 comp 边，扫描第二个非 comp 用双标签。

**结果**：BE021 20→28（+8 行超手册），130mm 边被路由到 p122 而非 p123。

**失败原因**：长度过滤无法可靠区分碎片归属——15/60/130 碎片的邻接数据相互污染。**根因归入瓶颈 1**：碎片→输出边映射丢失。BE021 p122/p123 需碎片级邻接检测，当前架构不支持。

## 架构瓶颈总结

### 瓶颈 1：3-SIDES 边收集流水线 ✅ 已大部分解决

碎片合并后丢失了碎片→输出边的映射关系。导致需 per-edge 判断的逻辑无法精确执行。

**已修复**：
- 合并函数 `_merge_collinear_edges` 返回 `[(length, op, source_fragments)]` 保留碎片
- 碎片记录 `nb_start`/`nb_end` 原始邻接数据（remap 前保存）
- 未标签构件体识别（匹配 `comp_dims` 尺寸）防止错误 remap
- 双端点碎片（起终点不同 neighbor）自动丢弃
- 碎片级标签覆写替代全局扫描，避免 BE023 类标签全错问题

**剩余问题**：BE021 p122/p123 双标签仍需碎片级双端邻接检测（当前架构仍假设单邻接格式）。

### 瓶颈 2：WM 线选择（`choose_weld_line`）

当 WM 箭头附近的零件有多条 LINE 时，`choose_weld_line` 按线长最短选择——但这未必是焊缝线。

**修复前提**：需在匹配层（`find_parts_at_point`）区分"端点命中"和"内部命中"的质量。

### 瓶颈 3：DXF 数据源限制

| 缺失类型 | 影响构件 | 行数 |
|----------|---------|:--:|
| p100 无 WM | CO007 | ~6 |
| p125/p127 无 WM | CO007/CO008 | ~4 |
| sp23/p169 工厂预制 | CO010 | ~55 |

这些缺口**非代码缺陷**——DXF 转换文件本身就不包含这些焊缝的任何几何信息。

## 已知遗留问题

| 构件 | 缺口 | 性质 |
|------|------|:--:|
| CO007 | p100[313]、p125[220]、板→板90.5mm | DXF 无对应 WM |
| CO008 | p127[60,90.5]、板→板90.5mm | DXF 无对应 WM |
| CO009 | hf/长度 ~8行 | per-edge 判断粒度 |
| CO010 | 3-SIDES self-ref 偏多 + 口径 ~55行 | 构件体 gusset + 缺子装配图 |
| BE021 | p123 156.5→155 1% 绘图公差 | DXF 几何精度 |

### CO010 — 统计口径差异（非代码缺陷）

经逐行对比分析，CO010 的 67 行缺口源于手册与脚本的**统计口径不同**：

| 手册口径 | 脚本口径 |
|----------|----------|
| **全构件**焊缝（工厂预制 + 现场组装） | 仅单张组装 DXF 上的 WeldMark |
| 包含 sp23/p169 等工厂预制零件（sp23 不在当前 DXF 的 BOM 或标签中） | 仅处理当前 DXF 可见的零件和 WM |
| p182/p183 互焊（12 行）等子装配焊缝 | p183 WM 仅指向 CO010，无子零件互焊 |
| 按 hf 等级拆分（如 hf=9 vs hf=12） | 按 WM 标注的 hf 值记录 |

**结论**：CO010 的 105/166 不是代码缺陷。约 55 行差额涉及工厂预制阶段的焊缝，需 CO010 的**子装配/零件图**对应的 DXF 文件才能覆盖。
